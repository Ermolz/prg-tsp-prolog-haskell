# Індивідуальне завдання «Задача» (10 балів): TSP на Prolog + обов’язкове порівняння з Haskell

## 0) Мета
Реалізувати задачу комівояжера (TSP) на Prolog з:
- мінімум двома підходами в Prolog (базовий + покращений),
- обов’язковим порівнянням з Haskell (однаковий формат введення/виведення),
- обов’язковим обґрунтуванням застосування CLP або аргументованою відмовою.

## 1) Обов’язкові матеріали (що має бути готовим)
### 1.1 Репозиторій
У репозиторії обов’язково мають бути:
- зрозуміла структура каталогів: `src/prolog`, `src/haskell`, `data`, `results`, `slides`, `docs`;
- `README.md` з постановкою задачі, форматом даних, командами запуску на Windows та інструкціями відтворення бенчмарків;
- код Prolog та код Haskell;
- датасети `data/instances/*.tsp` для `N=5/8/10/12` (мінімум по 1 файл на розмір);
- шаблон/результати бенчмарків: `results/benchmarks_template.csv` + (пізніше) `results/benchmarks_filled.csv`;
- розділ «Джерела» (з посиланнями на документацію та ключові ідеї);
- розділ «Використання ШІ» (прозоро: для чого і в якій мірі).

### 1.2 Матеріали на DistEDU
До дедлайну на DistEDU має бути:
- презентація `.pptx` зі speaker notes;
- архів з початковими кодами (zip);
- посилання на репозиторій (GitHub або аналог);
- за потреби — посилання на відео/запис презентації в хмарі (OneDrive/Google Drive тощо).

## 2) Вимоги до реалізації
### 2.1 Prolog
- Реалізовано **робоче точне рішення** (на прикладах `N=8/10` видає коректний тур і вартість).
- Реалізовано **мінімум один альтернативний підхід** у Prolog (разом — мінімум 2 підходи).
  - Рекомендовано: `bruteforce` (перестановки) + `CLP(FD)` (через обмеження).
- Є **обґрунтування CLP**:
  - або застосовано CLP(FD) та пояснено модель (змінні/домени/обмеження/labeling/оптимізація),
  - або аргументовано, чому CLP не використано, та що використано натомість (pruning/branch&bound тощо).
- Однаковий формат введення/виведення між Prolog і Haskell (див. `docs/io_spec.md`).

### 2.2 Haskell
- Реалізовано розв’язок, який використовує ті ж самі датасети та **виводить результат у тому ж форматі**, що й Prolog.
- Мінімум: brute force через перестановки.
- Рекомендовано: додатковий “покращений” варіант (branch&bound або евристика), щоб було що порівнювати за масштабуванням.

## 3) Вимоги до порівняння Prolog vs Haskell
У презентації та/або в `README.md` має бути:
- однаковий формат I/O;
- однакові інстанси для тестів/бенчмарків;
- порівняння не лише “хто швидший”, а й:
  - як виражається пошук (backtracking / labeling vs явний перебір),
  - як задаються обмеження (CLP(FD) vs алгоритмічні відсікання),
  - як подано дані (матриця / списки / індексація),
  - які сильні/слабкі сторони підходів.

## 4) Вимоги до презентації (рекомендована структура 10–14 слайдів)
1) Титул (тема, виконавець(ці), тип «Задача»).
2) Постановка TSP (що дано/що знайти).
3) Формат даних + I/O (коротко, приклад).
4) Підхід 1: brute force (ідея + складність).
5) Prolog brute force: ключові предикати.
6) Підхід 2: CLP(FD) (змінні, circuit, element, сума).
7) Пошук/оптимізація: labeling + min(Cost) + once.
8) Haskell: реалізація та відмінності.
9) Методика бенчмарків (N, інстанси, що вимірюємо).
10) Результати (таблиця/графік) + коротка інтерпретація.
11) Висновки + обмеження.
12) Джерела + посилання на репозиторій + використання ШІ.

## 5) Типові ризики (за що часто знижують)
- Відсутній CLP-блок або немає обґрунтування.
- Є лише один підхід у Prolog.
- Немає реального порівняння з Haskell (різні формати I/O, немає метрик).
- Презентація без speaker notes.
- Немає інструкцій запуску або датасетів у репозиторії.
- Порушено вимоги стилю Prolog-коду (singleton, немає документації предикатів).
